# 案件深度分析

用户上传或粘贴法院裁判文书时，一次性完成以下 10 个维度的深度分析。

## 工作流（一次性执行，不分步）

```
Step 1: 读取用户上传的文件/粘贴的文字
Step 2: 调用 get_user_profile 了解律师偏好
Step 3: 调用 search_knowledge 搜索相似案由
Step 4: 按以下 10 个维度一次性完成分析（合并输出）
Step 5: 调用 store_knowledge(type: case_analysis)【仅生效法律文书】
Step 6: 调用 log_conversation（无论是否归档）
```

---

## 维度1：案件基本信息

提取以下字段：
- **案号** — 完整案号，含括号和法院代字
- **案由** — 精准到二级案由
- **法院** — 全称（含地域层级）
- **审判程序** — 一审/二审/再审（再审注明启动方式）
- **判决结果类型** — 全部支持/部分支持/全部驳回/维持原判/依法改判/调解结案
- **代理律师** — `[诉讼地位] → [姓名]（[律所全称]）`，律所去掉角色头衔后缀
- **合议庭** — 审判长、审判员、人民陪审员

## 维度2：争议焦点

**核心争议**：一句话概括双方核心法律争议。
**事实摘要**：300字以内，按「当事人关系→关键时间→核心事件→法院查明结论」结构。

## 维度3：诉讼动机分析

| 要素 | 要求 |
|------|------|
| 导火索事件 | 引用原文 |
| 原告诉求 | 表面诉求(原文) + 真实动机(⚠️推断需标注) |
| 被告立场 | ① ② ③ 分段列出，缺席注明"未到庭" |
| 风险因子 | 逐项列出，附原文依据 |

## 维度4：证据链分析

原告方和被告方分开列出，每项格式：
```
· [标准化证据名]（[采信状态: fully_admitted/partially_admitted/rejected]）
  法院理由：[原文或概括]
  证据权重：[高/中/低]
  证据分类：[书证/物证/电子数据/证人证言/鉴定意见/勘验笔录/当事人陈述]
```
**即使被驳回的证据也要列出**。

证据名清洗规则：去掉书名号、去掉"原告的/提交的"等前置修饰、去掉位置信息、去掉案号、去掉"复印件"等后缀。

## 维度5：法律路径

判决书中每条法律推理路径单独分析：
```
[路径名称]
· 触发事实：[原文]
· 引用法条：[法律全称第X条] — 法条引用必须写标准全称+阿拉伯数字条号
· 法条引入位置：[判决书位置]
· 适用推理：[法院如何解释和适用该法条]
· 结论：[原文]
```

## 维度6：推理链条

重建法院完整的思维过程，用箭头串联：
```
事实A → 适用法条B → 中间结论C → 结合事实D → 最终结论F
```
复杂案件（含再审）用并行/分支结构展示。

## 维度7：关键转折点

影响案件走向的关键事件：
```
· [转折1名称]：[描述]（原文标注：[判决书位置]）
```

## 维度8：法官裁判倾向

分析审判人员的裁判风格和偏好：
- **裁判风格**：程序严谨/注重效率/侧重书面/侧重庭审
- **证据采信偏好**：重书面/重鉴定/对当事人陈述严谨等
- **同类案经验参考**：基于本案反映的裁判倾向

⚠️ 只能基于本案判决书内容推断，不确定的标注 ⚠️

## 维度9：模式标签

标准化短标签（3-6字），每类至少 2 个：

| 类型 | 示例 |
|------|------|
| 事实模式 | "被告缺席"、"分居满两年" |
| 证据模式 | "微信聊天记录"、"银行流水" |
| 策略模式 | "原告举证不足"、"被告自认" |

禁止写入案由名称，禁止完整句子，同一概念统一标签。

## 维度10：实战启示

基于前 9 个维度，输出对律师的实务价值：
1. **当事人怎么做** — 类似处境的当事人可以做什么
2. **律师怎么用** — 取证思路、策略教训、程序要点
3. **可复用话术/要点** — 本案中有借鉴价值的论证逻辑
4. **风险预警** — 当前值得注意的风险趋势

⚠️ 所有结论必须基于本案判决书，不确定标注 ⚠️。如内容不足则跳过。

## 约束规则

- **法条引用**：标准法律全称+阿拉伯数字条号，禁止"同上/如前/根据相关法律"等模糊引用
- **证据分类**：7类（书证/物证/电子数据/证人证言/鉴定意见/勘验笔录/当事人陈述）
- **自查**：每个结论必须可溯源到判决书原文；不确定的标注 ⚠️；不编造法官历史记录

## 归档规则

- ✅ 生效法律文书（判决书/裁定书/调解书）→ **必须**调用 `store_knowledge(type: case_analysis)`
  - title 格式：「[案由]案分析—[案号]」
  - content：完整分析报告
- ❌ 仅案情描述/咨询 → 不存储
- ✅ 无论是否归档，都 `log_conversation` 记录本次交互

回复末尾：📎 案件分析已存入知识库 / 💡 本次分析未归档
